How to name things consistently ‚Äî especially when nesting

Why Does Naming Matter?

When you come back to your code in a month (or someone else reads it), can you tell what .font_1 styles? What about .titlecontainer vs .title-container ‚Äî are those the same convention?

Inconsistent naming makes code hard to read, hard to maintain, and hard to debug. A system fixes this.

BEM: Block Element Modifier

BEM is the industry standard naming convention. Every element belongs to a block (the component), and is named flat relative to that block ‚Äî no matter how deeply nested in the HTML.

.block__element
card__header card__body card__title
nav__link nav__logo hero__title

The Rules

1. Lowercase + Hyphens

‚úì .card__header
‚úì .nav__menu-item
‚úó .cardHeader
‚úó .card_header

Lowercase always. Hyphens between words within a name. Double underscore separates block from element.

2. Describe Purpose

‚úì .card__title
‚úì .card__company-name
‚úó .font_1
‚úó .big-blue-text

Name what it IS, not what it looks like. Styles can change; purpose doesn't.

3. Flat Elements

‚úì .card__title
‚úì .card__logo
‚úó .card__header__title
‚úó .card__header__logo

ALL elements are flat relative to the block. Never chain with multiple double-underscores.

The Key BEM Rule: Elements Are Flat

No matter how deeply nested your HTML is, all elements are named flat relative to the block. You never chain element names. This is the rule that prevents class name collisions.

The Wrong Way: Chaining Elements

Many beginners try to mirror the DOM structure in their class names. This gets out of control fast:

.card
.card__header
.card__header__logo ‚Üê ‚ùå WRONG ‚Äî never chain elements!
.card__header__title ‚Üê ‚ùå WRONG

The Right Way: Flat Elements

ALL elements belong directly to the block, no matter where they sit in the HTML:

.card ‚Üí the Block
.card__header ‚Üí element of card
.card__logo ‚Üí element of card (NOT of header!)
.card__title ‚Üí element of card
.card__subtitle ‚Üí element of card
.card__body ‚Üí element of card
.card__footer ‚Üí element of card

The class name .card__logo tells you everything you need: it's the logo element of the card block. You don't need to know it's inside the header ‚Äî that's a DOM detail, not a naming concern.

When to Create a New Block

If your block is getting too many elements (10+), or if a section could be reused elsewhere, consider breaking it into a new block:

.card ‚Üí main block
.card__header
.company-badge ‚Üí NEW block (reusable component)
.company-badge__logo ‚Üí element of company-badge
.company-badge__name ‚Üí element of company-badge
.card__body

Now .company-badge can be reused in other places ‚Äî in a sidebar, in a footer, wherever. It's its own block with its own elements.

Modifiers: Variations of Blocks or Elements

BEM also has modifiers for variants ‚Äî use double hyphens (--):

.card ‚Üí base block
.card--featured ‚Üí modifier: a featured card
.card--compact ‚Üí modifier: a compact card
.card__button ‚Üí base element
.card__button--primary ‚Üí modifier: primary button
.card__button--disabled ‚Üí modifier: disabled state

üîë BEM Naming Summary

  • Block: .card ‚Äî the standalone component
  • Element: .card__title ‚Äî a part of the block (double underscore)
  • Modifier: .card--featured ‚Äî a variation (double hyphen)
  • Never chain elements: .card__header__title is always wrong
  • Too complex? Break it into a new block instead of nesting deeper

Why BEM Solves the Collision Problem

Remember the problem you found? Two trees, both with an element named "body". With BEM, this is solved automatically:

‚ùå The Old Problem
/* Both elements have the same class! */
.body { ... }  /* Which body? */
.body { ... }  /* Collision! */
‚úì BEM Solution
/* Each element is prefixed with its block */
.card__body { ... }
.modal__body { ... }
/* No collision! */

Every element carries its block name. Even generic names like "body", "header", "title" become unique when prefixed with the block.

BEM Naming Flowchart

1. Is it a standalone component (card, nav, hero, modal)?
‚Üí It's a block: .card, .hero
2. Is it part of a block (no matter how deeply nested)?
‚Üí It's an element: .card__header, .card__title
3. Is it a variation of a block or element?
‚Üí It's a modifier: .card--featured, .card__button--primary
4. Is the block getting too complex (10+ elements)?
‚Üí Break it into smaller blocks

üîë The Golden Rules of BEM

  • Block: standalone component ‚Üí .card
  • Element: part of a block ‚Üí .card__title (double underscore)
  • Modifier: variation ‚Üí .card--featured (double hyphen)
  • Flat elements: .card__title NOT .card__header__title
  • The class name should tell you: what block it belongs to, and what it is
