How to Read Census Table Codes

A workflow-driven guide. Click any argument in the commands below to learn what it means and how to use it.

Quick workflow summary

1. Browse for your table
   → censusreporter.org to find the right table by topic

2. Get the exact variable code
   → vars <- load_variables(2022, "acs5", cache = TRUE) in R
   → search by keyword: vars %>% filter(str_detect(concept, "SNAP"))
   → or by table: vars %>% filter(str_detect(name, "B22001"))

3. Decide: ACS or Decennial?
   → ACS for detailed topics (income, SNAP, education) — get_acs()
   → Decennial for basic population/race at block level — get_decennial()

4. Pick your year and geography, then pull
   get_acs(geography = "county", variables = "B22001_002", state = "MD", year = 2022)

How Census data gets made

Every year, the Census Bureau mails out the American Community Survey (ACS) to about 3.5 million households. It asks questions about income, housing, education, language, commuting, and more — including whether your household received SNAP benefits (food stamps). Every 10 years, the Decennial Census tries to count every single person, but asks far fewer questions.

Someone fills out the survey

A household in Prince George's County, MD checks "Yes" on: "In the past 12 months, did anyone in this household receive Food Stamps or SNAP benefits?"

The Census Bureau collects & aggregates

They combine everyone's answers by geography (state, county, tract, etc.) and organize them into standardized tables. Each table covers one topic and breaks it down by categories.

Tables get a code

Each table gets an ID like B22001. Each category within that table gets a number like _002. Together: B22001_002.

You pull it in R

Using tidycensus, you request specific variable codes at the geography and year you want.

The rest of this guide is organized around actual commands. Click any argument to expand its explanation.

get_acs( why not get_decennial()?

There are two main Census data products, and they use different functions:

American Community Survey
ACS

How often: Every year

Who's asked: ~3.5M households/year (a sample)

Topics: Hundreds — income, education, SNAP, commuting, disability, etc.

Smallest geo: Block group

Margin of error: Yes — always

Table prefixes: B, C, S, DP

R function: get_acs()

Decennial Census
Census

How often: Every 10 years (2000, 2010, 2020)

Who's asked: Everyone (full count)

Topics: Very few — population, race/ethnicity, age, housing occupancy

Smallest geo: Block

Margin of error: No (but has disclosure avoidance noise)

Table prefixes: P, H

R function: get_decennial()

Important: These use completely different table codes. You can't use B22001_002 in get_decennial() and you can't use P1_001N in get_acs(). Different products, different codes, different functions.

Quick decision: Need detailed topics like income, SNAP, education, commuting? → ACS. Just need total population or race at block level? → Decennial. (See Command 04 below for a Decennial example.)

geography = "county" all geography strings

Census geographies nest inside each other like containers:

Nation (1 — the whole US)
 └── Region (4 — Northeast, Midwest, South, West)
    └── Division (9 — e.g. South Atlantic)
       └── State (50 + DC + territories)
          └── County (~3,200)
             └── Census Tract (~85,000)
                └── Block Group (~240,000)
                   └── Block (~8 million — Decennial only)

What is a census tract? A tract is a semi-permanent statistical area drawn by the Census Bureau inside each county, designed to contain roughly 1,200 to 8,000 people (ideally about 4,000). They're meant to approximate neighborhoods — in a dense city a tract might be a few blocks, while in a rural area it could cover many square miles. They don't follow city boundaries, zip codes, or school districts — they're their own thing. Tracts stay mostly stable over time (they only get redrawn after a Decennial Census if the population has changed significantly), which makes them useful for tracking changes in a neighborhood over the years. A block group is a subdivision within a tract (usually 600–3,000 people), and a block is the smallest unit — typically a single city block bounded by streets.

What are ZCTAs? (and why not just "zip codes")

Zip codes are mail routes maintained by USPS — they can cross county and state lines, change constantly, and don't represent areas (some are just a single building). ZCTAs (Zip Code Tabulation Areas) are the Census Bureau's best attempt to turn zip codes into geographic areas. They're close but not identical to zip codes. Pull them nationally (no state argument) and filter by GEOID prefix:

get_acs(geography = "zcta", variables = "B22001_002", year = 2022) %>%
  filter(str_starts(GEOID, "207")) # 207xx = College Park area
What are congressional & legislative districts?

Congressional districts are the 435 U.S. House districts, redrawn after each Decennial Census. Each state gets at least one. Maryland has 8. The GEOID is 4 digits: state (24) + district number (01–08).

State legislative districts are for state legislatures — each state has its own system:

StringWhat it isMaryland example
"state legislative district (upper chamber)"State senate47 districts
"state legislative district (lower chamber)"State house/assembly/delegates141 districts

Nebraska is unicameral — it only has an upper chamber. The naming convention comes from the Census, not your state's specific terminology.

What are CBSAs?

CBSA = Core Based Statistical Area. These are metro and micropolitan areas defined by the Office of Management and Budget. A metro area is a core city of 50,000+ plus surrounding counties tied by commuting patterns (e.g. "Washington-Arlington-Alexandria, DC-VA-MD-WV"). A micropolitan area has a core of 10,000–49,999.

CBSAs cross state lines, so you don't pass a state argument — just pull them nationally:

get_acs(geography = "cbsa", variables = "B22001_002", year = 2022)
What are "places"?

In Census terminology, a place is any incorporated city, town, village, or borough, plus Census-Designated Places (CDPs) — unincorporated communities that the Census gives boundaries to for statistical purposes (like "Columbia, MD" which isn't technically a city). Places don't nest neatly into counties — a city can span multiple counties.

get_acs(geography = "place", variables = "B22001_002", state = "MD", year = 2022)
School districts — three types

Some states have a single school district per area (unified). Others split elementary and secondary into separate districts. You need to know which type your state uses:

StringUse when
"school district (unified)"Most states — one district covers K–12
"school district (elementary)"States that split elementary / secondary
"school district (secondary)"States that split elementary / secondary

Maryland uses unified districts. States like California and Illinois have all three types. If you're not sure, try unified first — if it returns no results, try the others.

Watch out: Smaller geographies = bigger margins of error. A tract-level estimate might have a MOE nearly as large as the estimate itself. Always check the moe column.

The category number — what "rows" really mean

When the Census says a table has 7 "rows," they mean 7 categories — 7 different things being measured. Not 7 rows of data like 7 counties.

Tables are hierarchical. Parent categories equal the sum of their children. Here's table B22001:

_001 Total households
├── _002 Received SNAP ← parent
│ ├── _003 Has someone 60+ ← child
│ └── _004 No one 60+ ← child
└── _005 Did NOT receive SNAP ← parent
├── _006 Has someone 60+
└── _007 No one 60+

The math: _002 + _005 = _001. And _003 + _004 = _002.

How to find codes: load_variables()

vars <- load_variables(2022, "acs5", cache = TRUE)

This gives you a dataframe with three columns:

ColumnWhat's in itExample
nameThe variable codeB22001_002
labelDescription with !! nestingEstimate!!Total:!!Household received Food Stamps/SNAP...
conceptThe table's topicRECEIPT OF FOOD STAMPS/SNAP...

If you know the table number (e.g. from Census Reporter):

vars %>% filter(str_detect(name, "B22001"))

If you don't know the table, search by keyword:

# What tables mention SNAP?
vars %>% filter(str_detect(concept, "SNAP")) %>% distinct(concept)

# Drill deeper
vars %>% filter(str_detect(concept, "SNAP"), str_detect(concept, "POVERTY"))

Heads up: Labels are ALL CAPS and use official Census phrasing. Try "FOOD STAMPS" instead of "SNAP". Use regex("snap", ignore_case = TRUE) for case-insensitive search. Or run View(vars) in RStudio for a searchable spreadsheet.

geography = "congressional district" geography strings with spaces

Some geography strings have spaces or parentheses. You have to type them exactly. Here are the tricky ones with code examples:

# Congressional districts
get_acs(geography = "congressional district", ...)

# State senate districts
get_acs(geography = "state legislative district (upper chamber)", ...)

# State house/assembly districts
get_acs(geography = "state legislative district (lower chamber)", ...)

# School districts
get_acs(geography = "school district (unified)", ...)

# Block groups (also has a space!)
get_acs(geography = "block group", ...)

# ZCTAs — pull all nationally, then filter
get_acs(geography = "zcta", variables = "B22001_002", year = 2022) %>%
  filter(str_starts(GEOID, "207")) # MD-ish zip codes
variables = c(snap_with_senior = "B22001_003", ...) naming & multiple variables

By naming the variables (snap_with_senior = "B22001_003"), the output uses your labels instead of raw codes in the variable column. This makes the output much easier to read.

The result has two rows per district — one for snap_with_senior and one for snap_no_senior. Because _003 and _004 are children of _002 in the tree, their sum equals the total SNAP households.

You can also pull a whole table at once using table = "B22001" instead of variables. That gets all 7 variables. (See Command 03 for an example.)

→ Two rows per district (one for each named variable)

get_acs(

table = "B22001" whole table vs. specific variables

Instead of listing individual variables, you can pull every variable in a table at once with table. For B22001 that's all 7 categories. Useful when exploring a table for the first time — you can always filter down later.

You use either table or variables, not both.

county = "033" filtering to a single county

When pulling tracts or block groups, you can optionally filter to one county using its 3-digit FIPS code. "033" = Prince George's County. Without this, you'd get every tract in the entire state.

You can find county FIPS codes with the fips_codes dataset built into tidycensus, or by searching "Prince George's County FIPS."

Wide format is better when you want to do math across variables (like mutate(pct_dogs = B99001_002E / B99001_001E)) because the values you need are in the same row. Use whichever makes your analysis easier.

survey = "acs1" 1-year vs. 5-year (not shown above — optional arg)

This command doesn't include survey, so it defaults to the 5-year ("acs5"). But you can switch to the 1-year by adding survey = "acs1". Same function, same table codes — just different data:

survey = "acs5"
ACS 5-Year

Data from: 5 years pooled ("2022" = 2018–2022)

Sample size: ~17.5M households

Available for: Every geography including tracts and block groups

Margins of error: Smaller (more data = more precise)

Currency: Less current — blends 5 years

Best for: Small areas, tract-level, most research

survey = "acs1"
ACS 1-Year

Data from: A single year ("2022" = just 2022)

Sample size: ~3.5M households

Available for: Only 65,000+ population areas (~800 counties)

Margins of error: Larger (less data = less precise)

Currency: More current — a single year

Best for: Timeliness, trends in large areas

The tradeoff: Think of it like a camera. The 5-year is a long exposure (smoother, blurrier in time) and the 1-year is a quick snapshot (sharper in time, but noisier and can only see bright things).

When the 1-year will fail: If you request tracts, block groups, or rural counties — the 1-year simply doesn't have data. Most individual counties don't qualify (only ~800 of ~3,200). If you need anything below county level, use the 5-year.

For change-over-time: The 1-year is cleaner because each year is independent. With 5-year, "2018" (2014–2018) and "2022" (2018–2022) share 2018, muddying the comparison. But you can only use the 1-year if your areas are big enough.

Rule of thumb: Start with the 5-year (the default). Only switch to 1-year if you specifically need single-year timeliness and your geographies are large enough.

The Decennial Census

Command 04 — Decennial population count

Total population by county, 2020 Census

get_decennial(
  geography = "county",
  variables = "P1_001N",
  state = "MD",
  year = 2020
)
get_decennial( different function, different codes

The Decennial Census uses a completely different function and completely different table codes from the ACS. You cannot mix them — B22001_002 doesn't exist in the Decennial, and P1_001N doesn't exist in the ACS.

The Decennial is a full population count (not a sample), so it has no margin of error column — the output has a value column instead of estimate and moe. It covers very few topics (population, race, age, housing occupancy) but goes all the way down to the block level.

Available years in tidycensus: 2000, 2010, 2020.

variables = "P1_001N" Decennial table codes

Decennial codes look different from ACS codes. Notice the N suffix:

PrefixMeaningExample
P1Total population by raceP1_001N = total population count
P2Hispanic/Latino origin by raceP2_002N = Hispanic or Latino
P3Race for population 18+P3_001N = total population 18+
P4Hispanic/Latino by race, 18+P4_002N = Hispanic or Latino, 18+
H1Housing occupancyH1_001N = total housing units

These codes are from the 2020 PL redistricting file. The DHC (Demographic & Housing Characteristics) file has additional tables with a different numbering scheme.

To browse Decennial codes, use different dataset strings in load_variables():

load_variables(2020, "pl") # 2020 redistricting data
load_variables(2020, "dhc") # 2020 Demographic & Housing
load_variables(2010, "sf1") # 2010 Summary File 1
05

Comparing change over time

There's no "give me multiple years" argument. You pull each year separately, stack them, and pivot.

Command 05 — Multi-year comparison

SNAP households by county: 2015 vs 2018 vs 2022, each year in its own column

# Pull each year, tag it
snap_2015 <- get_acs(
  geography = "county", variables = "B22001_002",
  state = "MD", year = 2015
) %>% mutate(year = 2015)

snap_2018 <- get_acs(..., year = 2018) %>% mutate(year = 2018)
snap_2022 <- get_acs(..., year = 2022) %>% mutate(year = 2022)

# Stack → pivot → rejoin names
snap_wide <- bind_rows(snap_2015, snap_2018, snap_2022) %>%
  select(GEOID, year, estimate) %>%
  pivot_wider(names_from = year, values_from = estimate, names_prefix = "snap_") %>%
  left_join(snap_2022 %>% select(GEOID, NAME), by = "GEOID")
mutate(year = 2015) why tag each pull?

get_acs() doesn't include a year column in its output — it just gives you GEOID, NAME, variable, estimate, moe. Adding mutate(year = 2015) tags each row so you know which pull it came from after you stack them together.

Each year's pull looks like this (simplified to 2 counties):

GEOIDNAMEvariableestimatemoeyear
24033Prince George's County, MarylandB22001_00248,200±2,1002015
24031Montgomery County, MarylandB22001_00228,300±1,8002015
bind_rows() stacking dataframes vertically

bind_rows() glues the three dataframes on top of each other into one tall dataframe:

GEOIDNAMEvariableestimatemoeyear
24033Prince George's...B22001_00248,200±2,1002015
24031Montgomery...B22001_00228,300±1,8002015
24033Prince George's...B22001_00244,500±2,0002018
24031Montgomery...B22001_00226,800±1,7002018
24033Prince George's...B22001_00241,100±1,9002022
24031Montgomery...B22001_00225,100±1,6002022

Each county now appears 3 times — once per year. The background colors show the three original dataframes glued vertically. This long format is great for ggplot but hard to compare across years in a table.

pivot_wider() years as columns

pivot_wider(names_from = year, values_from = estimate, names_prefix = "snap_") reshapes so each year value becomes a column name:

GEOIDNAMEsnap_2015snap_2018snap_2022
24033Prince George's County, Maryland48,20044,50041,100
24031Montgomery County, Maryland28,30026,80025,100

The three rows for Prince George's (one per year) got collapsed into one row with three columns. Calculate change: mutate(change = snap_2022 - snap_2015)

Why not include NAME in the pivot? The NAME column can have tiny formatting differences across ACS years — an extra space, different punctuation. If that happens, pivot_wider treats them as different rows and you get NAs. Using only GEOID as the ID and joining NAME back afterward is safer.

Caution with 5-year comparisons: The 5-year estimates are rolling averages. "2022" = 2018–2022 and "2018" = 2014–2018 — those overlap by 2 years. For truly non-overlapping comparisons, space years at least 5 apart (e.g. 2017 vs 2022).
