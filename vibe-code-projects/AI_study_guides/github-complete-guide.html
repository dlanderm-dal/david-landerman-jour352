The Big Picture

Before diving into buttons and commands, let's understand what GitHub actually is and why it exists. If you only read one tab, read this one.

What is GitHub?

GitHub is like Google Docs for code. Just like Google Docs lets multiple people work on the same document, track changes, leave comments, and see who wrote what -- GitHub does all of that for code files. It stores your files in the cloud, remembers every change ever made, and lets multiple people collaborate without overwriting each other's work. (docs)
Imagine you have a folder on your Mac Desktop called "my-website". You make changes to files inside it every day. Now imagine that folder could remember every version of every file you ever saved, let you jump back to any previous version instantly, and automatically sync to the internet so your teammates could see your work. That's GitHub.

What is Git vs GitHub?

Git = the engine under the hood Git is a program that runs on your computer. It tracks changes to files. It was invented in 2005 by the person who created Linux. Git works entirely on your own machine -- it doesn't need the internet. Git is not a programming language or a library -- it is a standalone software program (written in C) that you install and run as a command-line tool, much like how Finder is a program for managing files but Git is a program for managing file versions. (docs)
GitHub = the website that uses Git GitHub is a website (github.com) built on top of Git. It takes everything Git can do and adds a visual interface, cloud storage, collaboration features, and social features like profiles and followers. Think of Git as the car engine and GitHub as the car with the dashboard, steering wheel, and GPS. GitHub.com provides a visual web interface for Git -- instead of typing commands like git commit in a terminal, you can click buttons on the website to do the same things. GitHub Desktop is a Mac app that offers the same convenience. In total, there are three ways to use Git: (1) Terminal commands, (2) the GitHub.com website, and (3) the GitHub Desktop app.

Git is a distributed version control system. This means every developer has a complete copy of the entire project history on their own computer -- not just the latest version. If the server goes down, any developer's copy can restore the whole project.

In practice, this means that when you clone a repository, you download the entire project and its full history to your machine. Every commit, every version, every branch reference -- it all lives in a hidden .git folder inside your project folder. If you lose internet, you can still commit changes, view history, and create branches locally. When you reconnect, you push your local commits to GitHub and everything syncs up. Git is smart enough to merge your offline work with whatever happened on the server while you were away.

What is a Repository?

A repository ("repo") is a project folder with superpowers. It looks and acts like a regular folder on your Mac -- it has files and subfolders inside it. But it also has a hidden layer that remembers every change you've ever made. Every save point. Every version. It's a folder with a built-in time machine. (docs)
Open Finder on your Mac. Navigate to any folder. That's basically what a repository looks like on your computer -- just a folder with files. The only difference is that inside that folder, there's a hidden folder called .git that stores all the version history. You'll never need to open or edit that hidden folder. It's Git's brain.

What Tools Do You Need?

IDE = Integrated Development Environment An IDE is your workspace for writing code, just like Microsoft Word is your workspace for writing documents, or Photoshop is for editing images. An IDE is a program where you open files, type code, and save them.
VS Code (Visual Studio Code) Free, made by Microsoft, and the most popular IDE in the world. It looks like a text editor but with features like color-coded text, file browsing, and a built-in Terminal. If someone says "open it in VS Code," they mean open the folder/file in this app.
Cursor Cursor is VS Code but with AI built directly into it. It looks identical to VS Code, uses all the same features, but adds the ability to chat with an AI about your code and have it write/edit code for you. It's essentially "VS Code + ChatGPT in one app." By comparison, VS Code with an AI extension (like Claude, GitHub Copilot, or Microsoft Copilot) gives you a chat sidebar and inline suggestions, but Cursor goes further -- the AI is woven into every part of the editor, able to read your entire project, edit multiple files at once, and understand the full context of what you're building. Think of VS Code + Copilot as having a helpful assistant sitting next to you, while Cursor is more like the assistant has their hands on the keyboard. The Tools & Setup tab explores these options in more detail.
Same as VS Code, plus an AI chat panel on the right
GitHub Codespaces VS Code running in your web browser. You don't install anything -- you just go to a GitHub repository, click a button, and a full IDE opens in your browser tab. Great for getting started quickly or when you're on someone else's computer. Because Codespaces is VS Code running in your browser, the buttons, menus, and shortcuts are identical to the desktop version. The Tools & Setup tab walks through the VS Code interface in detail. (docs)
VS Code running inside your browser -- no installation needed
GitHub Desktop (not an IDE -- it's a visual Git client) A simplified visual app made by GitHub. It's not for writing code -- it's for managing your Git operations (committing, pushing, pulling, switching branches) without using the Terminal. Think of it as a visual remote control for Git. You use it alongside an IDE, not instead of one. GitHub Desktop handles the most common Git operations -- commit, push, pull, branch, merge, view history, and resolve conflicts -- covering roughly 95% of what beginners need. Some advanced operations like rebasing or cherry-picking still require the terminal. (docs)
You don't need to understand code to use GitHub. Think of it as a really powerful folder system with a time machine.

Every Button and Tab Explained

⬇ Code Button (Green) The big green button in the top-right. This is how you download or clone the repository to your computer. Clicking it shows options: "Open with GitHub Desktop", "Download ZIP", or a URL to use with the Terminal. This does NOT edit anything -- it just downloads.
📄 Code Tab Shows you the files in the repository, like looking at a folder in Finder. This is the default view when you open any repo. You can click on folders to navigate inside them and click on files to see their contents.
⚠ Issues Tab A built-in bug tracker and discussion board. Anyone can open an "issue" to report a bug, request a feature, or ask a question. Think of it like a to-do list for the project. Each issue can be assigned to someone, labeled, and discussed. (docs)
🔀 Pull Requests Tab This is where proposed changes live. When someone makes changes on a branch and wants to merge them into the main version, they open a "pull request" (PR). It's like saying "Hey, I made some edits -- can you review them before we combine them into the real version?"
▶ Actions Tab Automated tasks that run when certain things happen. For example: "Every time someone pushes code, automatically check for errors." You probably won't use this as a beginner, but it's good to know it exists.
⚙ Settings Where you configure the repository: rename it, make it public/private, enable GitHub Pages, manage who can access it, and more. Only the repo owner (or people with admin access) can change settings.
🍴 Fork Button Makes a complete copy of someone else's repository and puts it in YOUR GitHub account. It's like photocopying someone's entire folder so you can make changes to your copy without affecting theirs. Your fork retains a connection to the original repo (GitHub shows "forked from..." at the top), but changes don't automatically sync between them -- you must manually pull updates from the upstream repo or submit pull requests back to it. (docs)
☆ Star Button Bookmarks the repository. Starring a repo doesn't do anything functional -- it's just like adding a bookmark in your browser. You can find all your starred repos later from your profile.
👁 Watch Button Subscribes you to notifications for this repo. If someone opens an issue, makes a commit, or creates a pull request, you'll get notified. You can choose to watch "all activity," "releases only," or "nothing."
🌴 Branch Dropdown Shows which branch you're currently viewing and lets you switch to a different one. By default, you're looking at the "main" branch. (More on branches in the Branches tab.)
Commit History You can find this by clicking "X commits" near the top of the file list. It shows a timeline of every save point (commit) ever made, who made it, and when. It's the "time machine" view of your project.
Forks are copies. Branches are parallel timelines. They look similar but serve different purposes. A fork copies the entire repo to your account. A branch creates a parallel version within the same repo.

You repeat this loop over and over as you work. Edit, save, commit, push. That's it.

Let's break down each step:

What is a Commit?

A commit is a save point with a label. When you save a file (Cmd+S), you're saving it on your computer. But Git doesn't know about that save yet. A commit tells Git: "I want to record the current state of my files as a checkpoint." Each commit includes a message describing what you changed. (docs)
Think of it like "Save As" in Microsoft Word. When you save a Word document, you overwrite the old version. But if you use "Save As," you create a new copy with a new name. A commit is like using "Save As" every time, except Git does it automatically and keeps all the old versions behind the scenes. You don't end up with "project_v1.html", "project_v2_final.html", "project_v2_FINAL_REAL.html" -- Git handles all of that cleanly.

Good Commit Messages

# Descriptive and specific:
"Add navigation bar to homepage"
"Fix broken image link on about page"
"Update footer phone number"
"Remove unused CSS from stylesheet"

Bad Commit Messages

# Vague and unhelpful:
"stuff"
"update"
"fix"
"asdfasdf"
"final version"

What is Push?

Push = upload your commits to GitHub. Your commits only exist on your computer until you push them. Pushing sends your commits to GitHub.com so they're stored in the cloud and visible to your team. No push = nobody else can see your work.
You know how Finder has files that only exist on your laptop? If you drag them to Google Drive or iCloud, they upload and sync. Pushing is like dragging your work to the cloud. Until you push, it's only on your machine.

What is Pull?

Pull = download changes others made. If your teammate pushed new commits to GitHub, those changes exist on GitHub but NOT on your computer yet. Pulling downloads those changes to your local machine so you're up to date. It's the opposite of push.

What is Clone?

Clone = download a repository for the first time. When you find a repository on GitHub.com and want to work on it on your own computer, you "clone" it. Cloning downloads the entire repository -- all files, references to all branches, all history -- to a folder on your Mac. (Note: only the default branch is checked out locally; other branches exist as references you can switch to.) After cloning, that folder IS the repository, and you can start editing. (docs)
Prerequisite: You need Git installed on your computer. Most Macs come with Git pre-installed. You can check by opening Terminal and typing git --version.
Cloning is like downloading a ZIP file from the internet and unzipping it into a folder, except the folder also has a connection back to the original source so you can push and pull changes.

What is Fork?

Fork = make your own copy of someone else's repository. Forking happens on GitHub.com (not on your computer). When you fork a repo, GitHub creates a duplicate of it under YOUR account. You now own that copy and can make whatever changes you want without affecting the original. It's commonly used in open-source projects where you want to contribute to someone else's project.
Fork vs Clone -- What's the Difference? A fork happens on GitHub.com and creates a copy in your online account. A clone downloads a repo to your computer. Typically you fork first (to get your own copy on GitHub), then clone your fork (to get it on your computer).

What This Looks Like in Practice

In GitHub Desktop (Visual App)

Edit a file in VS Code or any editor and save it
Open GitHub Desktop -- it automatically detects your changes
Type a commit message in the "Summary" box at the bottom left
Click the blue "Commit to main" button
Click "Push origin" in the top bar to send it to GitHub

In Terminal (Command Line)

# Step 1: Check what changed
$ git status

# Step 2: Stage your changes
$ git add .

# Step 3: Commit with a message
$ git commit -m "Add nav bar"

# Step 4: Push to GitHub
$ git push
The Staging Area

Before committing, you must "stage" your changes with git add. Think of it like putting items in a box before sealing it. You choose exactly which changes go into each commit. The period in git add . means "everything that changed." You can also add specific files: git add index.html. (docs)

Commit early, commit often. Each commit is a save point you can go back to. If you make 100 changes and only commit once, you can only undo all 100 at once. If you commit after each change, you can undo them one at a time.

Branches & Merging

Branches are one of Git's most powerful features. They let you work on different versions of your project at the same time without breaking anything.

What Are Branches?

A branch is a parallel timeline of your project. When you create a branch, you're making a copy of your project at that moment in time. You can make changes on the branch without affecting the original. When you're happy with the changes, you merge the branch back. If the changes were bad, you just delete the branch. (docs)
Imagine duplicating a folder in Finder: "my-website" becomes "my-website-experiment." You make changes in the experiment copy. If the changes work, you replace the original folder with the experiment. If they don't, you trash the experiment. That's what branches do -- except Git does it more efficiently without literally duplicating all your files.
main vs master

The default branch is called main on new repositories, but older repos may use master. They work the same way -- only the name is different. You can configure the default branch name in your GitHub settings.

Visual Branch Diagram

Reading the Diagram

The green line is the "main" branch -- the real, live version. The blue line is a feature branch where someone is building a navbar. The purple dot is the merge point -- where the navbar work gets combined back into main. Notice that work continued on main (the "other work" commit) while the feature was being built separately.

Why Branches Exist

Branches let you experiment without risk. If you're working on a team, you don't want to make changes directly to the main version that everyone depends on. What if your change breaks something? Branches let you work in isolation. The main branch stays clean and working while you experiment on your branch.

What Shows Up on GitHub Pages?

GitHub Pages serves the main branch by default. When you enable GitHub Pages (free website hosting), it looks at the main branch and publishes whatever HTML files are there. Changes on other branches are invisible to your live website until they're merged into main. This is useful -- you can work on a new feature branch without your live website changing.

What is a Merge?

Merging combines two branches together. When you're done working on a branch and everything looks good, you merge it into main. Git takes all the changes from your branch and applies them to main. After merging, main has everything it had before PLUS your new changes.

What is a Pull Request?

A Pull Request (PR) = asking permission to merge. Instead of merging directly (which could introduce bugs), you open a Pull Request on GitHub. A PR says: "Here are the changes I made on my branch. Can someone review them before we merge into main?" Your teammates can look at your changes, leave comments, suggest edits, and then approve the merge. (docs)
The PR Review Process 1. You push your branch to GitHub.
2. You click "New Pull Request" on GitHub.com.
3. You write a description of what you changed and why.
4. Reviewers look at your code and leave comments.
5. Once approved, someone clicks "Merge pull request."
6. Your branch's changes are now part of main.
Branch Protection Rules

On team projects, administrators can protect the main branch so nobody can push to it directly. All changes must go through a pull request with code review. This prevents accidents. (docs)

Merge Conflicts

A merge conflict = two people edited the same line. If you changed line 15 of index.html on your branch, and your teammate also changed line 15 on their branch, Git doesn't know which version to keep. It flags this as a "conflict" and asks you to manually decide. Don't panic -- conflicts are normal and fixable.
What a conflict looks like in a file: Git adds markers showing both versions. You delete the markers and keep the version you want (or combine both).
<<<<<<< HEAD
<h1>Welcome to My Site</h1>
=======
<h1>Welcome to Our Amazing Site</h1>
>>>>>>> feature/update-heading

In this example, HEAD is your version (main), and feature/update-heading is the incoming branch's version. You pick which heading to keep, delete the conflict markers, save, and commit.

Branch Naming Best Practices

Pattern Example When to Use
feature/ feature/add-login Building something new
fix/ fix/broken-nav Fixing a bug
update/ update/footer-links Updating existing content
experiment/ experiment/new-layout Trying something you might throw away
Think of branches like making a copy of a document to try edits. If you like them, you merge it back. If not, you delete the copy. The original stays safe the whole time.

Local vs Remote

One of the most confusing things for beginners is understanding where your files actually live. Let's clear that up.

The Two Copies

💻

The files on your Mac. You edit them here. Git tracks changes here. This is where you do your actual work.

You interact with it via:
  • Finder (to browse files)
  • VS Code / Cursor (to edit files)
  • GitHub Desktop (to commit/push)
  • Terminal (for Git commands)

Upload your commits
← PULL
Download commits from other collaborators on GitHub
← CLONE
First-time download

Remote (GitHub.com)

The cloud copy at github.com. Your team sees this version. This is the "source of truth."

You interact with it via:
  • Web browser (github.com)
  • GitHub Desktop (push/pull)
  • Terminal (git push / git pull)
Think of it like a Google Doc. You have a copy on your computer (local) and a copy in Google Drive (remote). If you edit offline, your changes only exist on your computer until you go online and sync. Similarly, your Git commits only exist on your machine until you push them to GitHub.

What is localhost?

localhost = your computer pretending to be a web server. When you see a URL like http://localhost:3000, that's your own computer serving a website to itself. Nobody else on the internet can see it. It's like a private preview of your website.
Why does localhost exist? Some websites need a server to work properly (for example, sites that load data from databases or use JavaScript frameworks). Your computer runs a temporary mini-server so you can preview those sites while developing them.
The number after the colon is the "port." localhost:3000 means "my computer, door number 3000." Different projects use different ports so they don't conflict. It's just an address.
Does localhost update instantly? Most local development servers include "live reload" or "hot reload" — they watch your files and automatically refresh the browser when you save a change, making them even faster than file:// (where you have to manually press Cmd+R). To start a localhost server, open Terminal, navigate to your project folder with cd ~/my-project, and run a command like python3 -m http.server 8080 (for a basic server) or npm start (for JavaScript projects). The server runs until you press Ctrl+C or close the Terminal window. With VS Code's "Live Server" extension, you just right-click an HTML file and select "Open with Live Server" — no Terminal needed.

What is file:// ?

file:// = opening an HTML file directly, with no server. When you double-click an HTML file in Finder, your browser opens it with a URL like file:///Users/yourname/my-site/index.html. This means the browser is reading the file straight from your hard drive. There's no server involved. For simple HTML/CSS pages, this works fine.
file:// vs localhost vs a live URL These are three different ways to view a website, from simplest to most "real."
Method URL Looks Like Server Running? Who Can See It?
file:// file:///Users/you/site/index.html No Only you
localhost http://localhost:3000 Yes (local) Only you
Live URL https://yourname.github.io Yes (GitHub's server) Everyone on the internet

What happens when two people edit the same file? When two people edit the same file at the same time, Git detects a "merge conflict" when you try to push or pull. Git marks the conflicting lines in the file with special markers so you can see both versions:
<<<<<<< HEAD
Your version of the line
=======
Their version of the line
>>>>>>> main
You choose which version to keep (or combine both), delete the markers, and commit the resolved file. This is covered in more detail in the Branches & Merging tab. Circular references (where two branches depend on each other's changes) are rare in practice — Git's branching model prevents them by design. Conflicts are normal and not dangerous — they just need a human decision.

Branches Exist in Both Places

Each branch exists both locally and remotely (if you push it). When you create a branch on your computer, it only exists on your computer. When you push that branch, it also exists on GitHub. Pulling a branch someone else created downloads it to your computer. You can think of each branch as having two copies -- one on your machine, one in the cloud -- and you keep them in sync with push/pull.

What This Looks Like in Finder and GitHub Desktop

In Finder

The files you see in Finder always reflect whichever branch you currently have "checked out." If you switch branches in GitHub Desktop or Terminal, the files in Finder literally change. Don't be alarmed -- Git is swapping which version you see. All versions are safely stored.

In GitHub Desktop

GitHub Desktop has a "Current Branch" dropdown at the top. Clicking it shows all your local branches and any remote branches. Switching branches here instantly updates the files in Finder. You'll also see "Push origin" or "Fetch origin" buttons that sync your local and remote copies.

Your local copy and the GitHub copy are like two copies of a Google Doc. You have to sync them manually with push and pull. They don't auto-sync. If you forget to push, your teammates can't see your work. If you forget to pull, you're working on an outdated version.

GitHub Pages

One of GitHub's best free features is the ability to turn any repository into a live website.

What is GitHub Pages?

GitHub Pages = free website hosting from a repository. You put HTML, CSS, and JavaScript files in a repository, flip a switch in settings, and GitHub gives you a URL where the site is live. No server to manage, no hosting fees. It's completely free. (official docs)

How It Works

Your repo's HTML files become a live website. GitHub reads the HTML files in your repository and serves them as a real website that anyone can visit. If your repo has an index.html file, that becomes the homepage.
Which branch gets served? Usually the main branch. You can configure this in Settings. Any changes you push to main are automatically deployed to the live site (usually within a minute or two).
What URL do you get? The default URL format is: https://YOUR-USERNAME.github.io/REPO-NAME/. For example, if your username is "jdoe" and your repo is "my-portfolio", the URL would be https://jdoe.github.io/my-portfolio/.
Two types of GitHub Pages sites: Personal Site vs. Project Sites Your Personal Site is your one homepage per GitHub account. You create a repo named exactly yourusername.github.io, and it appears at https://yourusername.github.io. You only get one. Project Sites are attached to individual repos — any repo can have one, and you can have unlimited Project Sites. A repo called my-portfolio would live at https://yourusername.github.io/my-portfolio. Personal Sites are great for portfolios; Project Sites are great for individual assignments or experiments.

How to Enable GitHub Pages

Go to your repository on GitHub.com
Click the Settings tab (gear icon)
Scroll down to the "Pages" section in the left sidebar (under "Code and automation")
Under "Source", select "Deploy from a branch"
Under "Branch", select "main" and "/ (root)"
Click "Save"
Wait 1-2 minutes, then visit https://YOUR-USERNAME.github.io/REPO-NAME/
What gets published?

Every HTML file in your main branch becomes a page on your site. index.html is the homepage. about.html would be at /REPO-NAME/about.html. Subfolders work too: if you have projects/gallery.html, it becomes /REPO-NAME/projects/gallery.html.

Updating Your Live Site

Every push to main automatically updates the site. Once Pages is enabled, the workflow is simple: edit your files locally, commit, push, and the live website updates automatically. There's no separate "deploy" step. Push = publish.

Your Learning Hub on GitHub Pages

This Learning Hub could be hosted on GitHub Pages. Since the Learning Hub is just HTML, CSS, and JavaScript files in a repository, enabling GitHub Pages would give it a public URL that anyone can visit. Each study guide (including this one) would be accessible at its own URL.
GitHub Pages turns your repository into a live website. For free. Every push to main automatically updates what visitors see. It's the simplest way to publish a website.
Important Limitation

GitHub Pages only hosts static sites -- HTML, CSS, and JavaScript. It cannot run server-side code like Python, PHP, or Node.js backends. If your project needs a server, you'll need a different hosting service.

Tools & Setup

GitHub doesn't live in isolation. There's an ecosystem of tools you'll encounter. Let's demystify them one by one.

Homebrew (brew)

Homebrew is a tool installer for Mac. It's like the App Store, but for developer tools. Instead of downloading a .dmg file from a website and dragging it to Applications, you type a command in Terminal and Homebrew installs it for you. Developers use it because it's faster and keeps everything organized.
Prerequisite: Terminal To use Homebrew, you need to understand Terminal. Terminal is an app on your Mac (find it in Applications > Utilities) that lets you type commands instead of clicking buttons. It's a text-based way to talk to your computer. When someone says "run this in Terminal," they mean: open Terminal, type the command, and press Enter.
# Installing Homebrew (you only do this once):
$ /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# Installing a tool with Homebrew:
$ brew install node

# That's it. Node.js is now installed on your Mac.
Think of Terminal as talking to your Mac through text messages instead of tapping on icons. And Homebrew is like texting "install Node.js" to your Mac and it just does it.

Node.js

Node.js is a runtime environment — a program that can execute JavaScript code outside of a web browser. JavaScript is the programming language that makes websites interactive (animations, buttons, forms). Normally, JavaScript ONLY runs inside a web browser (Chrome, Safari, Firefox). Before Node.js (created by Ryan Dahl in 2009), JavaScript could only run inside browsers. Node.js uses Chrome's V8 engine to run JavaScript on your computer directly, without needing a browser at all.
Prerequisite: Understanding JavaScript's normal home JavaScript was invented to run inside web browsers. When you visit a website and see a dropdown menu animate or a form validate your email, that's JavaScript running in your browser. For a long time, that was the ONLY place JavaScript could run.
So why does Node.js matter? Node.js freed JavaScript from the browser. Now developers use JavaScript to build servers, command-line tools, and desktop apps. This matters because it enabled an entire ecosystem of developer tools: package managers (npm), build tools (webpack, Vite), server frameworks (Express), and testing tools — all written in JavaScript. Even if you never write JavaScript yourself, you'll encounter Node.js because tools like VS Code extensions, React, and many website frameworks depend on it being installed.

npm (Node Package Manager)

npm = the official package manager that comes bundled with Node.js. When you install Node.js, you get npm automatically — npm (Node Package Manager) was created alongside Node.js and is maintained by GitHub (which is owned by Microsoft). It is the standard tool for downloading and managing JavaScript packages. npm lets you download pre-written JavaScript code (called "packages" or "libraries") that other people made, so you don't have to build everything from scratch. There are alternatives like Yarn and pnpm, but npm is what you'll use by default.
Prerequisite: What is a library? A library is pre-written code that someone else created and shared for free. Instead of writing 500 lines of code to create a date picker, you download a library that already does it. Libraries save enormous amounts of time. The JavaScript ecosystem has millions of them.
How people find libraries npmjs.com is the main registry -- it's like the App Store for JavaScript libraries. The npmjs.com registry is the largest software registry in the world, hosting over 2 million packages. You search for what you need (e.g., "date picker"), compare options, and install the one you want.
# Installing a library with npm:
$ npm install lodash

# This downloads the "lodash" library into your project.
# You can now use it in your code.

package.json

package.json = a recipe file listing what libraries your project needs. Every project that uses npm has a file called package.json in its root folder. A root folder is the top-level folder of your project — the main folder that contains everything else. If your project lives at ~/Documents/my-website/, that my-website folder is the root. When someone says "put this file in the root folder," they mean put it directly inside my-website/, not inside a subfolder like my-website/css/. This file lists every library the project depends on, along with version numbers. It's like a recipe listing ingredients. When someone clones your project, they run npm install and npm reads this file to download everything needed.
// A simplified package.json looks like this:
{
  "name": "my-project",
  "version": "1.0.0",
  "dependencies": {
    "lodash": "^4.17.21",
    "express": "^4.18.2"
  }
}

node_modules

node_modules = where downloaded libraries actually live. When you run npm install, npm creates a folder called node_modules and puts all the downloaded libraries inside it. You run npm install in the Terminal (or VS Code's built-in terminal, which opens at the bottom of the window). The key detail: npm installs packages into whichever folder your Terminal is currently "in." If your Terminal prompt shows ~/Documents/my-website, then npm install creates the node_modules folder inside my-website/. This is why you always cd (change directory) into your project folder before running npm commands — the packages go into the project, not your whole computer. Each project has its own separate node_modules. This folder can be HUGE (hundreds of megabytes). Two critical rules:
Never edit anything inside node_modules. These files are managed by npm. If you change them, npm will overwrite your changes next time you install.
Never upload node_modules to GitHub. It's too large and completely unnecessary. Anyone who clones your project can recreate node_modules by running npm install, which reads package.json. That's why .gitignore exists.

.gitignore

.gitignore = a file that tells Git which files to NOT track. It's a plain text file in your project's root folder. Each line is a file or folder name that Git should ignore. When Git sees these files, it pretends they don't exist. They won't be committed, pushed, or tracked. (docs)
# A typical .gitignore file:

node_modules/
.DS_Store
.env
*.log
dist/

This is not written in any programming language — it's just a visual representation of how folders and files are organized on your computer, like what you'd see in Finder on your Mac. The indentation shows what's inside what: css/ and js/ are folders inside my-project/, and style.css is a file inside the css/ folder.

my-project/
css/
style.css
js/
app.js
index.html
package.json
.gitignore
node_modules/ ← ignored by Git
.DS_Store ← ignored by Git
.env ← ignored by Git
Why .gitignore matters for API keys and secrets: The .env file in the example above is where developers store API keys and passwords. By listing .env in .gitignore, Git will never track or upload it. This is critical — if you accidentally push API keys to GitHub, they can be stolen within minutes by automated bots. See the Best Practices tab for a full explanation of protecting sensitive information.

GitHub Codespaces

Codespaces = a full VS Code editor in your browser, no installation needed. GitHub Codespaces gives you a complete development environment that runs in your web browser. It looks and works exactly like VS Code on your Mac, but everything runs on GitHub's servers. This is great for getting started quickly, working on someone else's computer, or avoiding setup headaches. GitHub gives you 60 hours of free Codespaces usage per month on a personal account -- more than enough for coursework. Your codespace pauses automatically after 30 minutes of inactivity, so you won't waste your hours.

How to Launch a Codespace

Go to any repository on GitHub.com
Click the green "Code" button (top right of the file list)
Click the "Codespaces" tab (next to "Local")
Click "Create codespace on main"
Wait for it to load (this may take 30 seconds to a minute the first time)
You're now in VS Code -- in your browser! You can edit files, use the terminal, install extensions, and push commits just like on your Mac.

Best Practices

Knowing how to use GitHub is one thing. Using it well is another. These practices will save you headaches and make your work professional.

Repository Hygiene

One repo per project, not per file. A common beginner mistake is creating a new repository for every single file. Instead, create one repository per project. A website is one repo. An app is one repo. A collection of related scripts is one repo. The repo contains all the files that belong together.
Archive old repos -- don't delete them. You might think you'll never need an old project again, but you'd be surprised. GitHub lets you "archive" repos -- they become read-only but are still accessible. This keeps your profile clean while preserving your work.
Use clear, descriptive repo names. Your future self (and anyone who visits your GitHub profile) should immediately understand what a repo contains from its name alone.

Good Repo Names

  • my-portfolio-site
  • weather-app
  • jour352-data-project
  • class-study-guides

Bad Repo Names

  • repo1
  • test
  • stuff
  • untitled
  • asdfjkl
Use README.md files. Every repo should have a README.md file in its root folder. This file is automatically displayed on the repo's main page on GitHub.com. It should explain: what the project is, how to set it up, and any other important info. It's written in Markdown (a simple text formatting language -- think of it as a simpler version of HTML).

Commit Message Discipline

Every commit message should answer: "What did this change do?" When you look at your commit history months from now, vague messages like "update" tell you nothing. Descriptive messages let you find exactly when and why something changed.
Message Why It's Good/Bad
✓ "Add navigation bar to homepage" Specific: tells you exactly what was added and where
✓ "Fix broken image link on about page" Clear problem and location
✓ "Update footer with new phone number" Describes the change and its purpose
✗ "stuff" Useless -- tells you nothing
✗ "update" Updated what? Everything is an update.
✗ "fix" Fixed what? Where? Why?
✗ "final version" Narrator: It was not, in fact, the final version.

Branch Discipline

Never commit directly to main on team projects. The main branch should always be stable and working. If you make changes directly to main and break something, everyone is affected immediately. Instead, create a branch, make your changes there, test them, then merge via a pull request.
Use feature branches for every change. Even small changes should get their own branch. This keeps main clean and makes it easy to review, undo, or isolate changes.
Delete branches after merging. Once a branch has been merged into main, delete it. It served its purpose. Leaving old branches around is like leaving finished drafts on your desk. GitHub even shows a "Delete branch" button after a merge.

.gitignore Best Practices

To ignore files, you create a plain text file called .gitignore in your project's root folder (the same level as your index.html or package.json). Each line in .gitignore lists a file or folder pattern that Git should pretend doesn't exist. Git will never track, stage, commit, or upload anything matching those patterns. You create it like any other file — in VS Code, right-click your project folder, select "New File," and name it exactly .gitignore (with the dot at the beginning, no file extension). Then type one pattern per line.
# Downloaded libraries — recreated with npm install
node_modules/

# Mac system files
.DS_Store

# Secret keys and passwords
.env

# Build output
dist/

The # lines are comments (notes to yourself) — Git ignores them.

Always ignore these files:
Pattern What It Is Why Ignore It
node_modules/ Downloaded JavaScript libraries Huge folder; anyone can recreate it with npm install
.DS_Store Mac Finder metadata file Invisible Mac file with no purpose in your project
.env Environment variables (passwords, API keys) Contains sensitive information that should never be public
*.log Log files Generated automatically; not part of your code
dist/ Build output folder Generated from source code; can be rebuilt
Warning

If you accidentally commit a .env file with passwords or API keys, those secrets are now in your Git history FOREVER -- even if you delete the file in a later commit. The history remembers. Always set up .gitignore before your first commit.

Protecting Sensitive Information

What are API keys? API keys are like passwords for online services. When your code needs to talk to an external service (like a weather API, a maps service, or a database), the service gives you a unique key — a long string of letters and numbers — that identifies you. If someone else gets your key, they can use (and abuse) the service on your account, potentially costing you money or exposing your data.
Why you NEVER put API keys in code that goes to GitHub. GitHub repositories are often public, meaning anyone on the internet can see every file. Even in private repos, you should keep secrets separate from code as a best practice. Bots constantly scan GitHub for exposed API keys — if you push one, it can be stolen within minutes.
How to use .env files to store secrets locally. A .env file is a plain text file in your project's root folder where you store sensitive values. Each line follows the format KEY_NAME=value. For example:
WEATHER_API_KEY=abc123xyz789
DATABASE_PASSWORD=mySecretPass
Your code reads these values at runtime (using libraries like dotenv for Node.js or python-dotenv for Python) instead of having the secrets written directly in your source code.
How .gitignore prevents secrets from being uploaded. By adding .env to your .gitignore file, Git will completely ignore it. The file stays on your computer but is never committed, pushed, or tracked. This means your API keys stay local and private. When someone else clones your project, they create their own .env file with their own keys. Many projects include a .env.example file (without real values) to show what keys are needed.
# Example .env file (NEVER commit this):

WEATHER_API_KEY=abc123xyz789
DATABASE_URL=postgres://user:pass@localhost:5432/mydb
SECRET_TOKEN=super_secret_value_here
# In your .gitignore, make sure this line exists:

.env

New Repo vs New Branch?

When to create a new repository: When the project is completely different and unrelated. A portfolio site and a data analysis project are separate repos. A class study guide and a personal blog are separate repos.
When to create a new branch: When you're working on a new feature or experiment within the same project. Adding a new page to your website = new branch. Trying a different color scheme = new branch. Fixing a bug = new branch.

New Repo

  • Completely different project
  • Different purpose / audience
  • No shared files with existing repos
  • Different collaborators

New Branch

  • New feature in existing project
  • Bug fix in existing project
  • Experiment you might throw away
  • Same codebase, different version
The best GitHub users have clean, well-named repos with descriptive commits. It's like keeping a tidy desk -- you can find everything, nothing is cluttered, and anyone who looks at it can understand your work.

1. Advanced Branching Strategies

Git Flow A structured branching model with designated branch types:
main — production-ready code. Only receives merges from release or hotfix branches.
develop — the integration branch. All feature branches merge here first.
feature/* — individual features, branched from develop, merged back to develop.
release/* — preparation for a new release. Bug fixes only, then merged to both main and develop.
hotfix/* — emergency fixes on production. Branched from main, merged to both main and develop.
Trunk-Based Development A simpler approach: everyone works on main (the "trunk") with very short-lived branches (hours, not days). Changes are small and merged quickly. Popular at companies like Google. Requires good automated testing.
AspectGit FlowTrunk-Based
ComplexityHigh — many branch typesLow — just main + short branches
Best forScheduled releases, large teamsContinuous deployment, small teams
Branch lifetimeDays to weeksHours to 1-2 days
RiskMerge conflicts from long-lived branchesRequires strong testing
Git Flow is like planning a road trip with multiple stops and rest areas. Trunk-based development is like driving straight to the destination with quick bathroom breaks.

2. GitHub Actions & Automation

What is CI/CD? Continuous Integration (CI) means your code gets automatically tested every time you push. Continuous Deployment (CD) means your code gets automatically deployed (published) when tests pass. Together, they eliminate the "it works on my machine" problem.
What are GitHub Actions? Automated workflows that run when something happens in your repo. "When someone pushes code, run the tests." "When a PR is merged to main, deploy the website." You define these workflows in YAML files inside a .github/workflows/ folder in your repo.
GitHub Actions is like setting up email filters — "when X happens, automatically do Y."
A simple workflow example This file would live at .github/workflows/test.yml: 
name: Run Tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm install
      - run: npm test
Common workflows
Run tests on every push
Deploy to GitHub Pages when main is updated
Check code style (linting) on pull requests
Auto-label issues based on content

3. Code Review

Why code review matters Code review catches bugs, shares knowledge across the team, and keeps code quality consistent. Most professional teams require at least one review before merging any PR.
How to review a PR
1. Read the description — understand what the PR is trying to do before looking at code.
2. Check the diff — the "Files changed" tab shows exactly what was added (green) and removed (red).
3. Leave inline comments — click the + icon next to any line to comment on it.
4. Submit your review — choose Approve, Request Changes, or Comment (no decision).
Writing helpful review comments
Be specific — "This variable name is unclear" is better than "fix this."
Suggest alternatives — "Consider using forEach instead of a for loop here."
Explain why — "This could cause a bug because X."
Praise good work — "Nice solution to this problem!" goes a long way.
CODEOWNERS file A file at .github/CODEOWNERS that auto-assigns reviewers. For example: *.js @team-lead means any JavaScript file change automatically requests a review from @team-lead.

4. Organizations & Teams

What's an Organization? A shared GitHub account for groups — companies, classes, or open-source projects. Repos belong to the organization, not a single person. If someone leaves the team, the repos stay.
Permission LevelWhat They Can Do
ReadView code and issues
TriageManage issues and PRs (but can't push code)

5. Advanced Git Commands

CommandWhat It DoesWhen to Use It
git rebaseMoves your branch's commits on top of another branch, rewriting history for a cleaner timelineWhen you want a linear history instead of merge commits
git stashTemporarily saves your uncommitted changes so you can switch branchesWhen you need to switch branches but aren't ready to commit
git cherry-pickCopies a single commit from one branch to anotherWhen you need just one specific change from another branch
git reflogShows a log of every action you've taken, even "deleted" commitsWhen you accidentally lose work — reflog can find it
git reflog is your undo button for everything. Even if you think you deleted a branch or lost commits, reflog almost certainly has them. Git rarely truly deletes anything.
Rebase vs Merge git merge creates a "merge commit" that ties two branches together — your history shows the branch existed. git rebase replays your commits on top of the target branch — your history looks like a straight line. Merge is safer (preserves history), rebase is cleaner (linear history). Most beginners should stick with merge.

6. Contributing to Open Source

Finding repos to contribute to
GitHub Explore (github.com/explore) — curated collections and trending repos.
"good first issue" label — many repos tag beginner-friendly issues. Search: label:"good first issue" language:javascript
Projects you already use — found a bug in a tool you use? That's a contribution opportunity.
The fork-and-PR workflow
1. Fork the repo (makes your own copy on GitHub)
2. Clone your fork to your computer
3. Branch — create a feature branch for your change
4. Code — make your changes, commit them
5. Push to your fork
6. PR — open a pull request from your fork to the original repo
7. Respond to review feedback and iterate
Important repo files to read before contributing
CONTRIBUTING.md — how the project wants you to contribute (coding style, PR process, testing requirements)
CODE_OF_CONDUCT.md — behavior expectations for the community
README.md — project overview, setup instructions, and usage
Your first open-source contribution doesn't have to be code. Fixing a typo in documentation, improving a README, or reporting a well-described bug are all valuable contributions.
