1. What Is Agentic Engineering?

Agentic engineering is a new way of building software. Instead of writing every line of code yourself, you direct AI agents -- tools like Claude Code, Cursor, and GitHub Copilot -- that can write, edit, and refactor code on your behalf. Your job shifts from being the person who types code to being the person who designs systems, reviews output, and makes decisions.

The old model: you are the bricklayer You open a blank file, type code character by character, run it, see an error, fix it, run it again. Every line passes through your fingers. Your speed is limited by how fast you can type and how much syntax you can remember.
The new model: you are the architect You describe what you want built. The agent writes the code. You review it, test it, and refine your instructions. Your speed is limited by how clearly you can communicate intent and how well you can evaluate output.
Common AI coding agents
Claude Code -- Anthropic's CLI agent. Reads your project files, understands architecture, makes multi-file edits. Works from the terminal.
Cursor -- An AI-native code editor (based on VS Code). Has chat, autocomplete, and agent modes built into the IDE.
GitHub Copilot -- GitHub's AI pair programmer. Suggests code inline as you type, plus a chat mode for larger questions.
Windsurf, Aider, Cline -- other agents in the ecosystem, each with different approaches to how they interact with your codebase.

The patterns in this guide apply regardless of which agent you use. The principles of clear communication, structured feedback, and systematic review are universal. Simon Willison, a prominent developer and writer on AI tooling, calls this "AI-assisted programming" and emphasizes that the human remains responsible for understanding and validating every change the agent makes.

Writing code is cheap now. Designing systems, reviewing output, and maintaining quality is where the human value is.

2. The Feedback Loop

Every interaction with an AI agent follows a loop. Understanding this loop is the single most important concept in agentic engineering, because it changes how you think about getting work done.

The Core Agent Loop
Prompt
→
Agent Acts
→
Review
→
Refine
↻
Step 1: Prompt You describe what you want. The more specific your description, the better the result. Include context (what files exist, what the architecture looks like), constraints (what NOT to do), and expected output format.
Step 2: Agent Acts The agent reads your prompt, reads relevant files, and produces code changes. It might create files, edit existing ones, run commands, or ask clarifying questions.
Step 3: Review You read what the agent produced. Does it do what you asked? Did it break anything? Did it follow the project conventions? This is the most critical step -- never skip it.
Step 4: Refine If the output is not right, you give feedback. "The button should be blue, not red." "You forgot to handle the error case." "This works but it's too verbose -- simplify it." Then the loop repeats.

Iteration matters more than getting it right the first time. A mediocre first prompt followed by three rounds of refinement almost always produces better results than spending twenty minutes crafting the "perfect" prompt. The agent learns from your feedback within the conversation.

Before and After: The Power of Specificity

Vague Prompt

# What you might type on your first try: Make me a login page.

The agent has no idea what technology you use, what it should look like, what happens after login, or where it fits in your project. It will guess -- and it will guess wrong.

Precise Prompt

# A much better version: Create a login page in React using our existing AuthContext from src/context/. Use the same styling pattern as the signup page (src/pages/Signup.jsx). Fields: email and password. On success, redirect to /dashboard. On failure, show the error message from the API response below the form. Do NOT create a new CSS file -- use the existing styles in global.css.

The agent knows the tech stack, where to find existing patterns, what fields to include, what happens on success and failure, and what NOT to do.

A good prompt is worth more than a good programmer. The skill of agentic engineering is not knowing syntax -- it is knowing how to describe what you want with enough precision that an agent can build it correctly.

3. Debug, Log, and Diagnose

When something goes wrong in agent-built code, the quality of your error messages determines how quickly the problem gets fixed -- by you or by the agent in the next iteration. This is Best Practice #1: every project should have a robust debug and logging strategy.

Why Agents Need Readable Errors

AI agents read error messages the same way you do -- as text. If an error message says Error: undefined, neither you nor the agent can figure out what happened. If it says Error: User.findById() received null -- expected a valid ObjectId string but got undefined. Check that the request includes a userId parameter., both you and the agent know exactly what to fix.

Good vs Bad Error Messages

Bad Error Messages

// Unhelpful -- tells you nothing throw new Error("Something went wrong"); // Generic -- which operation failed? catch (err) { console.log("Error"); } // Silent failure -- worst of all catch (err) { // do nothing }

Good Error Messages

// Specific -- names the operation and input throw new Error( `Failed to load user profile: ` + `userId="${id}" returned 404 from API` ); // Context-rich -- includes what was attempted catch (err) { console.error( `[saveComment] Failed to save:`, { commentId, text, error: err.message } ); }

Best Practice #3: Errors for People AND Agents

Every error message in your project should be interpretable by both a human reading a browser console and an AI agent reading the error text in a prompt. This means:

  • Name the function or operation that failed (e.g., [saveComment], [fetchUserProfile])
  • Include the input that caused the failure (e.g., userId="abc123")
  • Describe what was expected vs what happened (e.g., "expected 200 OK, got 404 Not Found")
  • Never swallow errors silently -- an empty catch block is a bug waiting to happen

The DOM Dump Technique

Sometimes visual debugging fails. The page looks fine but behaves wrong, or the agent cannot "see" the page at all (since agents work with text, not screenshots). The DOM dump technique solves this: you serialize the relevant part of the page's DOM as a text string and paste it into the conversation so the agent can analyze the structure.

// Dump the DOM of a specific element as text // Paste the output into your prompt for the agent console.log( document.querySelector('.problematic-section').outerHTML ); // For a compact version, strip whitespace: console.log( document.querySelector('.sidebar') .outerHTML.replace(/\s+/g, ' ') );

This gives the agent the actual HTML structure to analyze, which is far more useful than you trying to describe what the DOM looks like in words.

If an error message doesn't tell you AND the agent what went wrong, it's not an error message -- it's noise.

4. The Commenting and Change Tracking System

AI agents have no memory between sessions. Every time you start a new conversation, the agent starts fresh -- it does not remember what it built yesterday, what bugs it fixed last week, or what decisions were made about the architecture. This is why change tracking is not optional -- it is essential infrastructure.

Best Practice #2: The Commenting Mechanism

A commenting system lets you mark specific parts of your project that need changes and export those annotations as structured prompts. The Learning Hub itself uses this pattern: you highlight text, leave a comment, and export all comments as a batch prompt for Claude Code. The key insight is that comments are the bridge between "what the human sees on screen" and "what the agent receives as instructions."

Best Practice #6: The Judicious Change Log

Every significant change to a project should be recorded somewhere persistent -- a file on disk that survives session resets. This serves as the project's institutional memory.

Why changelogs matter Context windows reset. Sessions end. Agents forget. Without a written record of what was changed and why, you will repeat mistakes, rebuild features that already exist, and lose track of architectural decisions.
Architecture files as persistent memory An architecture file (like ARCHITECTURE.md) serves as the project's "brain." It tells every new agent session what the project is, how files connect, what conventions to follow, and what has already been decided. The first instruction in every prompt should be: "Read ARCHITECTURE.md before doing anything."
The prompt changelog pattern For each study guide or major file, keep a corresponding .prompt.txt file that records the original generation prompt plus dated addenda for every subsequent edit. This creates a complete history of how the file evolved. The Learning Hub stores these in AI_study_guides/prompts/.
Example: Prompt Changelog Entry
## 2025-03-15 — User comment batch ## Changes requested: - Added comparison table for Git vs GitHub - Expanded the "What is a branch?" section - Fixed broken anchor link in sub-nav ## Methodology note: - User prefers inline expansion over callout boxes
An agent without a changelog is an agent that repeats mistakes. Invest in your project's persistent memory and every future session starts smarter.

5. Reuse, Don't Rebuild

One of the most common mistakes in agentic engineering is asking the agent to build something from scratch when a perfectly good solution already exists. This applies to libraries, existing features in your project, and patterns you have already established.

Best Practice #4: Judicious Use of Libraries

The JavaScript ecosystem (and most programming ecosystems) has millions of pre-built packages available through package managers like npm. Before asking an agent to build a date formatter, a color picker, a markdown parser, or a chart library, check whether a well-maintained package already does what you need.

Best Practice #5: Don't Remake Existing Features

This one is subtle. Within your own project, you may already have a component, function, or pattern that does exactly what the agent is about to build from scratch. Agents do not always know that your project already has a formatDate() utility in utils/helpers.js. You need to tell them.

Decision Flowchart: Build vs Reuse

Question If Yes If No
Does this exact feature already exist in your project? Use it. Point the agent to the file. Continue to next question.
Is there a well-maintained library that does this? Install it. Tell the agent to use it. Continue to next question.
Is this a common pattern with known best practices? Give the agent a reference implementation to follow. Continue to next question.
Is this truly unique to your project? Now build it from scratch -- with clear specs. You probably missed something above. Search again.
Cross-Reference: npm and package.json

If you are not familiar with npm (Node Package Manager) and how JavaScript projects manage dependencies, see the GitHub Complete Guide in this Learning Hub. It covers package.json, node_modules, and how libraries are installed and managed in JavaScript projects.

Before asking an agent to build something, ask: does this already exist? The fastest code is code you never have to write.

6. The "-ilities" -- Non-Functional Quality

When most people think about software quality, they think about "does it work?" But working code is the bare minimum. Best Practice #7 asks you to plan for all relevant quality measures -- the set of "-ilities" that determine whether software is actually good, not just functional.

Each of these qualities is something you should be thinking about when you review agent-generated code. The agent will not think about them unless you ask.

Quality ("-ility") One-Sentence Definition Example What the Agent Should Do
Accessibility Can everyone use it, including people with disabilities? Screen readers can navigate the page; images have alt text; forms have labels. Use semantic HTML, add ARIA attributes, ensure sufficient color contrast.
Testability Can you verify it works without manually clicking through everything? Each function can be tested independently; no hidden side effects. Write pure functions where possible; separate logic from UI; add test hooks.
Reliability Does it work consistently, even under stress? The app handles network failures gracefully instead of crashing. Add error handling, retry logic, fallback states, and loading indicators.
Security Can bad actors exploit it? User input is sanitized; API keys are not exposed in client code. Sanitize inputs, use environment variables for secrets, validate on the server side.
Maintainability Can someone else understand and modify this code in six months? Variables have clear names; functions do one thing; architecture is documented. Use descriptive names, add comments for "why" (not "what"), follow project conventions.
Observability Can you see what is happening inside the system when it runs? Logs show which API calls were made; errors include context; performance can be measured. Add structured logging, include request IDs in error messages, log state transitions.
Scalability Does it work with 10x more users or data? The search feature works with 100 items and with 100,000 items. Use pagination, avoid loading everything into memory, index database queries.
Usability Is it intuitive for the person using it? Buttons look clickable; error messages tell users what to do next; the layout makes sense. Follow platform conventions, provide feedback for every action, use clear labels.

You do not need to optimize for all eight qualities on every project. A quick internal tool might only need reliability and usability. A public-facing application needs all of them. The point is to consciously decide which qualities matter, rather than discovering gaps after launch.

Most bugs are not in the code that runs -- they are in the qualities nobody tested. An agent will build you a feature that works. It is your job to make sure it works well.

7. Security Audit Mindset

Best Practice #8 is the security audit. AI agents generate code that works, but they do not inherently understand what "safe" means in your context. An agent will happily hardcode an API key, accept unsanitized user input, or create a form that submits data over an unencrypted connection -- unless you specifically tell it not to.

Common Vulnerabilities in Agent-Generated Code

Cross-Site Scripting (XSS) When user-provided text is inserted directly into HTML without sanitization, a malicious user can inject scripts that run in other users' browsers. Agents often use innerHTML instead of textContent -- always check.
Exposed API Keys Agents sometimes place API keys, database credentials, or secret tokens directly in source code. If that code is pushed to a public GitHub repository, those secrets are exposed to the entire internet.
SQL Injection When user input is concatenated directly into a database query string, a malicious user can modify the query to read, modify, or delete data they should not have access to.
Missing Input Validation Agents often trust that input will be well-formed. They may not check for empty strings, overly long inputs, unexpected data types, or values outside expected ranges.

The .env Pattern

Secrets (API keys, database URLs, private tokens) should never appear in your source code. Instead, store them in a .env file at the root of your project, add .env to your .gitignore so it is never committed to version control, and access the values through environment variables in your code.

# .env file (NEVER commit this) API_KEY=sk-abc123xyz789 DATABASE_URL=postgres://user:pass@localhost/mydb # .gitignore (tells Git to ignore the .env file) .env node_modules/
// In your code, read from environment variables: const apiKey = process.env.API_KEY; // NEVER do this: const apiKey = "sk-abc123xyz789"; // EXPOSED!

Security Review Checklist for Agent-Generated Code

  • No API keys, tokens, or passwords hardcoded in source files
  • .env is listed in .gitignore
  • User input is sanitized before being inserted into HTML (use textContent, not innerHTML)
  • Database queries use parameterized statements, not string concatenation
  • Forms submit over HTTPS, not HTTP
  • Authentication tokens are stored securely (httpOnly cookies, not localStorage)
  • Error messages do not leak internal system details to end users
  • File uploads are validated (type, size, extension) before processing
  • Dependencies are up to date (run npm audit periodically)
Agents do not understand secrets. They will happily commit your API key to a public repo. Every piece of agent-generated code needs a security review -- even if it is just running through the checklist above.

8. The Placeholder Strategy

Best Practice #9 is placeholder text and incremental implementation. AI agents have context limits -- they can only process a certain amount of text at once (measured in "tokens," which are roughly word-sized chunks). If you ask an agent to build a 3,000-line page in one go, it will hit its limit partway through and either truncate the output, lose coherence, or start repeating itself.

The Roadmap-First Approach

Instead of asking for everything at once, build in passes:

Pass 1: Outline Ask the agent to create the full file structure with all sections, headings, and navigation -- but with placeholder text like "Content for this section will be added in a future pass." This establishes the skeleton.
Pass 2: Fill in sections Go section by section: "Now fill in section 3 with the content about debugging. Here is the existing file -- only modify section 3." Each pass is small enough to stay within context limits.
Pass 3: Iterate and refine Review each section, leave comments, and refine through the feedback loop. Fix issues section by section rather than rewriting the whole page.

Example: Building a Large Page

# Pass 1 prompt: Create a study guide page with 8 sections. Set up all the HTML structure, tab navigation, and section headings. For section content, put placeholder text: "TODO: Content for [section name]". Follow the structure in ARCHITECTURE.md. # Pass 2 prompt: Read the file you just created. Fill in ONLY section 3 ("Debug, Log, and Diagnose") with full content. Do not touch other sections. # Pass 3 prompt: Now fill in section 4 ("Change Tracking"). Do not modify sections 1-3 or 5-8.
Don't ask the agent to build a mansion in one go. Give it the blueprints, let it pour the foundation, then add rooms one at a time. Each pass stays focused, stays within context limits, and produces reviewable output.

1. Prompt Engineering for Agents

Prompt engineering is the skill of writing instructions that produce the output you want. For AI coding agents, this means structuring your requests so the agent has enough context to act correctly without enough rope to go off the rails.

Anatomy of a Good Agent Prompt

A well-structured prompt has four parts:

1. Context Setting Tell the agent what files to read, what the project architecture looks like, and what already exists. The Learning Hub uses a "FIRST: Read ARCHITECTURE.md" pattern at the top of every prompt to ensure the agent starts with the right mental model.
2. Task Description Clearly state what you want done. Be specific about inputs, outputs, and behavior. "Add a login form" is vague. "Add a login form with email and password fields that posts to /api/auth/login and displays server-side validation errors" is precise.
3. Constraint Specification Tell the agent what NOT to do. This is often more important than what you ask it to do. "Do NOT create a new CSS file -- use the existing styles." "Do NOT modify the database schema." "Do NOT install new dependencies."
4. Output Format Expectations Tell the agent what the result should look like. "Follow the same pattern as the existing UserCard component." "Use the comparison-table class from the style guide." "Include a Key Insight callout box at the end of each section."

The "FIRST: Read..." Pattern

The Learning Hub's exported prompts always begin with an instruction to read the architecture file. This pattern is broadly useful for any project:

# Start of every prompt: FIRST: Read ARCHITECTURE.md in the project root. It tells you what every file does, how the systems connect, and what conventions to follow. THEN: [your actual task here]

This ensures the agent has the project's full context before it starts making changes. Without it, the agent operates blind and is more likely to violate conventions or duplicate existing functionality.

Writing Effective Constraints

Weak Constraint Strong Constraint Why It Matters
"Keep it simple" "Use no more than 3 files and 0 new dependencies" Agents interpret "simple" differently than you do.
"Follow best practices" "Follow the naming pattern in src/components/ -- PascalCase for components, camelCase for utilities" Best practices vary. Point to your project's specific conventions.
"Don't break anything" "Run the existing tests after your changes and verify they still pass" Gives the agent a concrete verification step.
"Make it look good" "Use the same card layout as index.html with the .card-grid class" Points to a concrete visual reference.
The best prompts are not the longest. They are the ones that give the agent exactly enough context to succeed and exactly enough constraints to avoid going off track.

2. Version Control Integration

Git is your safety net when working with AI agents. Because agents can make sweeping changes across multiple files in seconds, the ability to revert to a known good state is not just useful -- it is essential.

The Branch-Per-Experiment Pattern

Before asking an agent to make a significant change, create a new branch. If the experiment works, merge it. If it does not, delete the branch and nothing is lost.

# Before a risky agent task: $ git checkout -b experiment/new-login-page # Let the agent make its changes... # Review the results... # If it worked: $ git checkout main $ git merge experiment/new-login-page # If it didn't work: $ git checkout main $ git branch -D experiment/new-login-page

Commit Early, Commit Often

After each successful agent iteration, commit the changes. This creates checkpoints you can return to. If the agent's third round of changes breaks something the second round got right, you can revert to the second commit without losing everything.

The rhythm Prompt the agent. Review the output. If it looks good, commit. Then prompt again for the next change. Each commit is a save point.

When to Revert vs When to Fix Forward

Situation Action Why
Agent made a small mistake in one file Fix forward -- tell the agent what to correct Faster than reverting and re-doing
Agent restructured multiple files incorrectly Revert to last good commit Untangling a multi-file mess is harder than starting over
Agent's approach is fundamentally wrong Revert and re-prompt with a different approach Iterating on a bad foundation wastes time
You're not sure if the changes are good Commit to a branch, test thoroughly, then decide Preserves the work while you evaluate
Treat Git as your "undo" button for agent work. Branches are cheap, commits are free, and the safety net they provide is priceless when an agent goes sideways.

3. Testing with Agents

Testing is where the architect mindset matters most. An agent can write code that looks correct but fails in edge cases you did not think to describe. A disciplined testing approach catches these gaps.

Red/Green TDD with Agents

Test-Driven Development (TDD) works surprisingly well with agents:

1. Write the test first (Red) Tell the agent: "Write a test that verifies a user can log in with valid credentials and gets redirected to /dashboard." The test will fail because the feature does not exist yet -- this is the "red" state.
2. Implement the feature (Green) Tell the agent: "Now implement the login feature so that the test passes." The agent has a clear target -- make the test green.
3. Refactor (Clean) Tell the agent: "The test passes. Now clean up the implementation -- remove any redundant code, improve naming, and ensure it follows our project conventions." The test acts as a safety net during refactoring.

The "Before AND After" Rule

Always run your test suite before the agent makes changes (to establish a baseline) and after (to verify nothing broke). Include this in your prompts:

# Include in your prompt: Before making any changes, run the existing tests to confirm they pass. After your changes, run them again and verify nothing broke. If any test fails after your changes, fix it before finishing.

Manual Testing Through Agent-Browser

For visual features that automated tests cannot easily cover, use agent-browser tools to take screenshots and verify the UI. The Learning Hub project uses agent-browser for this purpose -- a headless browser that can navigate to pages, take screenshots, and extract text.

Screenshot Verification

After the agent makes UI changes, take a screenshot and review it. This is especially important for CSS changes, layout modifications, and responsive design. The agent cannot "see" the page the way you can -- it works with code, not pixels.

Tests are the contract between you and the agent. If the tests pass, the agent delivered what you asked for. If they fail, something needs fixing. Without tests, you are flying blind.

4. Working with Context Limits

Every AI agent has a context window -- the maximum amount of text it can process in a single conversation. This is measured in tokens (roughly, one token equals one word or partial word). When a conversation exceeds the context limit, the agent starts "forgetting" earlier parts of the conversation.

How Context Windows Work

What counts as tokens Everything in the conversation: your prompts, the agent's responses, file contents the agent reads, and any system instructions. A 100,000-token context window sounds large, but reading three or four source files can consume half of it.
What happens when you hit the limit The agent begins dropping earlier messages from its working memory. It may "forget" instructions you gave at the start of the conversation, or lose track of changes it made earlier in the session.

The Architecture File as Persistent Memory

Because context windows reset between sessions, you need a way to carry knowledge forward. An architecture file (like ARCHITECTURE.md) serves this purpose. At the start of every new session, the agent reads the file and immediately has the full project context -- without needing you to re-explain everything.

Breaking Large Tasks into Agent-Sized Chunks

If a task requires reading or producing more text than the context window allows, break it into smaller pieces. Each piece should be self-contained enough that the agent can complete it in one session without needing to remember previous sessions.

The "if you haven't already" pattern When writing prompts that might be used across multiple sessions, include conditional instructions: "If you haven't already in this conversation, read ARCHITECTURE.md and the target file first." This prevents redundant reading if the agent already has the context, while ensuring it gets the context if it is starting fresh.
Context limits are not a bug -- they are a design constraint. Structure your work around them: keep sessions focused, use architecture files for persistent memory, and break large tasks into small passes.

5. Multi-Agent Workflows

As your projects grow in complexity, you may find that a single agent conversation is not enough. Multi-agent workflows use multiple AI sessions -- sometimes in parallel -- to tackle different aspects of a problem.

Using Subagents for Specialized Tasks

What is a subagent? A subagent is a secondary AI session spawned by the primary agent (or by you) to handle a specific subtask. For example, the main agent might spawn a subagent to research an API's documentation while it continues working on the implementation.
When subagents help Complex tasks where different parts require different kinds of expertise. One agent might be great at writing tests while another is better at designing UI components. Splitting the work lets each session stay focused.

Agent Orchestration Patterns

Pattern How It Works Best For
Sequential One agent finishes, its output becomes the next agent's input Pipeline tasks (generate code, then test it, then document it)
Parallel Multiple agents work on independent tasks at the same time Unrelated features that do not touch the same files
Supervisor One agent coordinates, delegating subtasks to specialist agents Complex projects where a "project manager" agent tracks overall progress
Review One agent generates, another agent reviews and critiques Quality-sensitive work where a second opinion catches mistakes

One Long Session vs Multiple Short Ones

One long session The agent accumulates context over the conversation. Good for tightly coupled changes where each step depends on the previous one. Risk: hitting context limits and losing early context.
Multiple short sessions Each session starts fresh with a focused task. Good for independent changes. Risk: losing continuity if the architecture file is not kept up to date.
The sweet spot Use one session per logical unit of work. A "logical unit" is a set of changes that makes sense together -- implementing a feature, fixing a bug, or refactoring a module. When the unit is complete, commit and start a new session for the next unit.
Multi-agent workflows are not about using more AI -- they are about using AI more effectively. Match the number and type of sessions to the shape of the problem.

6. Common Anti-Patterns

An anti-pattern is something that seems like a good idea at first but consistently leads to bad outcomes. Here are the most common mistakes people make when working with AI coding agents.

Anti-Pattern Why It Happens What to Do Instead
"Just fix it" (no context) You know what is broken, so you assume the agent does too. Always include: what the expected behavior is, what actually happens, and where the relevant code lives.
Rebuilding from scratch every session No architecture file or changelog, so each session starts blind. Maintain ARCHITECTURE.md and prompt changelog files. The agent reads these first and picks up where the last session left off.
Trusting agent output without review The code looks right at a glance. The agent seems confident. Read every change. Run the tests. Check for the -ilities. Agent confidence is not the same as agent correctness.
Ignoring error messages "It's probably not important." Or: "I'll fix it later." Paste the error into the conversation. Let the agent diagnose and fix it immediately. Unresolved errors compound.
Not using version control "It's just a small project." Or: "I'll set up Git later." Initialize Git before writing any code. Commit after every successful agent iteration. The cost of NOT having Git is one bad agent run away from losing hours of work.
Over-engineering simple tasks Agent suggests an elaborate solution; you accept it because it looks impressive. Ask yourself: "Is this the simplest thing that could work?" If a 5-line script solves the problem, you do not need a 200-line module with three abstraction layers.
The Most Dangerous Anti-Pattern

The single most dangerous habit is not reading what the agent produced. Speed is the whole point of AI-assisted development, but speed without review is recklessness. The agent is a powerful tool, but you are the one responsible for what ships.

Anti-patterns are easy to fall into because they feel efficient in the moment. The discipline of agentic engineering is doing the boring parts -- reviewing, testing, documenting -- even when the agent makes everything else fast.

This table summarizes the ten core best practices referenced throughout this guide. Use it as a checklist when starting a new project or reviewing an existing one.

# Best Practice One-Line Description
1 Debug / Logging / DOM Dump Build robust error reporting so both humans and agents can diagnose problems from the error text alone.
2 Commenting Mechanism Use a structured annotation system to mark what needs changing and export those annotations as agent prompts.
3 Errors for People AND Agents Every error message should name the operation, include the input, and describe expected vs actual behavior.
4 Judicious Use of Libraries Check for existing, well-maintained packages before asking the agent to build from scratch.
5 Don't Remake Existing Features Search your own project for existing components and utilities before letting the agent create duplicates.

These ten practices are not about slowing you down. They are about making sure the speed of AI-assisted development does not come at the cost of quality, security, or sanity. Fast and broken is worse than slow and solid.
